## Day 1 — 哈欠到底是什么信号

阅读了两篇铁路驾驶员疲劳检测论文（Pan Qing 高速铁路, Pan Xiaoyan 多模态融合）。技术路线很清晰：Dlib 面部 landmark → EAR/MAR 计算 → 疲劳分类。

但读完有一个直觉上的不适：**所有论文都把哈欠等价于危险信号**。Yawn = fatigue = warning → 强制干预。驾驶员没有选择空间。

人的哈欠不只是疲劳。人在舒适的座椅上、午饭后、长途高速巡航中都会哈欠。这些场景的共同点不是"危险"，而是**低唤醒度的放松状态**。

如果车里坐的是另一个人类，他看到你哈欠，会直接爆闪警告灯吗？不会。他可能会问："困了？要不要换个音乐精神一下？"或者什么都不说——取决于他对你状态的判断和你们之间的关系。

这个类比驱动了整个项目：**让车的行为更像一个观察你状态的副驾驶，而不是一个给你告警的传感器**。

## Day 2 — 搭建框架时的设计选择

构建了四层架构（Perception → Core → Interaction → Execution）+ 八状态 FSM。结构本身不特别，但几个设计选择值得记录：

**1. 为什么是 2 次哈欠触发而不是 1 次？**

1 次哈欠太容易误触——人打完一个哈欠可能立刻就意识到自己并不需要切换模式。2 次提供了一个最低限度的证据窗口：这不是一个偶发的哈欠，这是一个模式（pattern）。

这个设计的代价是响应延迟：用户可能哈欠完 2 次等了 30 秒系统才问。这是 sensitivity vs specificity 的 trade-off。当前版本偏向 specificity。

**2. 指数衰减评分**

大多数论文要求"连续 N 帧 MAR > threshold"才判定哈欠。这在真实场景中很脆弱——一次转头导致的 landmark 丢失就会让计数器归零。改用指数衰减：每一帧 MAR 低于阈值时，累积分数乘以 0.85，而非清零。这更接近人判断哈欠的方式：你不会因为一个人闭嘴 0.1 秒就认为他没在哈欠。

来源：e-candeloro/Driver-State-Detection 的 AttentionScorer 模型。

**3. 持续语音监听 vs 哈欠触发——两条路径的优先级**

持续监听（continuous voice listener）允许用户在任何时候说话切换模式。当它和哈欠触发流程冲突时，用户的主动指令赢了——FSM 立即重置回 MONITORING，取消正在进行的询问流程。

这个设计原则是：**用户主动操作优先级 > 系统发起流程**。这是 shared control 的一条具体实现规则。

## Day 3 — Debug 过程中暴露的真实问题

调试过程中几个 bug 的根因暴露了系统设计层面的薄弱环节：

**Bug: TTS 声音被麦克风捡回。** 根因是系统中没有区分"人说的话"和"系统自己说的话"。在真实车载环境中，TTS 通过独立声道输出，不会回环进麦克风。但在 PC 原型上，扬声器和麦克风共享物理空间。修复方式（2 秒冷却 + state gate）是 workaround 而非设计解决。这提醒我：**桌面原型与车规系统之间存在一条鸿沟，不仅是硬件，更是交互场景的根本差异**。

**Bug: 第一次语音指令需要 12 秒。** Whisper 模型在第一次 `transcribe()` 调用时才加载。这个延迟在 demo 中可以接受，但在真实车载场景中不可接受——驾驶员不会等 12 秒。工业化的解决路径：模型常驻内存（已在 `main.py` 中通过显式 `stt.load()` 做预热），或使用 streaming ASR 而非整句转写。

**Bug: 哈欠阈值不适用于个人。** MAR threshold 0.55 对我的面部偏高——我必须"卡通式大张嘴"才触发。不同人脸型、嘴唇厚度、甚至有无胡须都会影响 MAR 基线。正确的做法是 **per-user calibration**：系统启动时让用户做 30 秒的 neutral face + deliberate yawn，计算出个性化的基线 MAR 和阈值。当前实现没有做这一步。

## 反思：这个项目到底在做什么

回头审视，这个项目的核心不是"检测哈欠"，不是"语音识别"，也不是"状态机"。这些都是手段。

核心命题是：**在驾驶员状态感知系统中，如何设计 interaction 以尊重 user agency。**

具体来说：
- 感知信号（哈欠）存在根本性的意图模糊 → 不能用 if-then 逻辑覆盖
- 系统能做的是：观察 → 提出假设 → 向用户求证
- 用户回应之后，系统才执行

这是一种**询问式自动化（inquiry-based automation）**：系统的角色是 propose，不是 impose。

在 SAE L2/L3 驾驶员监控系统的语境中，这个方向有价值。当前的 DMS 大多是被动监控 + 告警。但这个项目试图走另一条路：**监控 + 理解 + 询问 + 执行**，其中理解环节不交给算法独自判断，而是让用户参与。

这个范式不限于哈欠→模式切换的场景。任何需要推断驾驶员意图的场景（导航偏好、音乐选择、车内温度）都可以套用这个模板：感知信号 → 状态线索 → 主动但非强制询问 → 用户确认后执行。

这不是一个"做好了"的项目。它是一组想法在可运行的代码中的一次试探。
