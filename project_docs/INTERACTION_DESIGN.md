# Interaction Design — Inquiry-Based Driver State Interaction

> 核心交互逻辑：系统如何在不剥夺用户控制权的前提下进行主动干预。

---

## 1. 两种模式

| | Dynamic Mode | Rest Mode |
|---|---|---|
| 场景意图 | 需要更活跃的驾乘体验 | 保持当前舒适状态 |
| 环境光 | 暖橙 (255, 80, 0) | 冷蓝 (80, 120, 255) |
| 虚拟空调 | 18°C / 强风 / 面部送风 | 26°C / 弱风 / 脚部送风 |
| 音效 | 高节奏 | 舒缓 |
| 触发条件 | 驾驶员回应 "Yes" 或主动说出 | 驾驶员回应 "No" 或喇叭触发后保持 |

两种模式不是简单的"运动 vs 舒适"二选一。它们代表了两种**用户意图路径**：一种是接受系统的主动建议（Dynamic），另一种是拒绝并维持现状（Rest）。Rest Mode 同时也是系统默认状态——这暗含了一个设计立场：**安静优先于打扰，维持现状优先于变更**。

## 2. 哈欠触发流程

```
驾驶员哈欠 2 次（Rest Mode 下）
  → FSM: MONITORING → YAWN_DETECTED → INQUIRING
  → TTS: "You seem relaxed. Would you like to change to Dynamic Mode?"
  → FSM: INQUIRING → LISTENING
  → 录音 + STT + NLP
  → 意图 = "dynamic": CONFIRMING → SWITCHING（切换到 Dynamic Mode）
  → 意图 = "rest":     CONFIRMING → SWITCHING（保持 Rest Mode）
  → 意图 = "unknown":  重试 2 次 → 放弃 → 回到 MONITORING
```

---

## 3. 为什么是"询问"而非"自动执行"

### 3.1 哈欠信号存在根本性的意图模糊

人哈欠的可能原因：
- 疲劳 / 低唤醒度 → 可能需要更刺激的驾驶环境
- 刚坐进车里感到舒适放松 → 不需要任何改变
- 午饭后的自然生理反应 → 不需要任何改变
- 长途驾驶中的拉伸 → 可能需要放松而非刺激

系统无法从 MAR 数值中区分这些场景。**任何单模态信号都不足以推断用户意图**。

### 3.2 自动切换的风险

如果系统看到 2 次哈欠就直接切到 Dynamic Mode：
- 用户可能正享受 Rest Mode 的安静氛围 → 系统行为构成打扰
- 用户可能不理解为什么座舱突然变了 → 信任受损
- 用户可能想切回 → 额外的操作负担

**自动执行 = 系统替用户做决定 = 剥夺 user agency。**

### 3.3 询问的 HCI 含义

询问式交互（inquiry-based interaction）的核心：
- 系统说："我观察到了 X。你想要 Y 吗？"
- 用户说："要"或"不要"

这个过程中：
- 系统承担**感知 + 建议**的职责
- 用户保留**决策 + 确认**的权力
- 交互是**双方协作**而非单方覆盖

这就是 **Shared Control** 在交互层面的具体实现。

---

## 4. Shared Control 与 Human-in-the-Loop

在自动化驾驶语境中，Shared Control 通常指方向盘/油门/制动的控制权分配。但本项目关注的是**座舱环境控制的共享**：系统可以感知、建议、准备执行，但**执行的前提是用户确认**。

Human-in-the-Loop 的体现：
- 每个状态转换的 gate 都是用户行为（哈欠、语音回应）
- 系统不预设用户意图，不跳过确认环节
- 出错（意图未知）时，系统回退到 Monitoring 而非强制执行

---

## 5. 双层交互路径

### Path A：哈欠触发（被动感知 → 主动询问）

驾驶员不需要主动操作。系统在后台持续观察，发现线索后主动发起对话。适合：
- 驾驶员专注于驾驶，不想手动操作
- 驾驶员自己未意识到状态变化（低唤醒度时人常常不自知）

### Path B：持续语音指令（主动控制）

驾驶员随时可以说 "switch to dynamic mode" 或 "switch to rest mode"。适合：
- 驾驶员明确知道自己想要什么
- 不想等待哈欠触发流程
- 在 Dynamic Mode 下想切回 Rest Mode（系统不会在 Dynamic Mode 下因哈欠触发询问）

两条路径**并存不互斥**：Path B 的执行会取消 Path A 的进行中流程（FSM 重置回 MONITORING）。用户主动指令的优先级高于系统发起的询问流程。

---

## 6. 非侵入式设计原则

1. **默认安静**：Rest Mode 是系统默认状态。不主动发声，不频繁询问
2. **可取消**：语音指令随时可用，不需要完成整个哈欠流程
3. **有限重试**：2 次意图理解失败后静默退出，不反复追问
4. **冷启动宽松**：需要 2 次哈欠才触发（不是 1 次），减少误触发
5. **状态门控**：Continuous voice listener 在 MONITORING 状态下才接受指令，避免与哈欠流程冲突
6. **Cooldown**：每次语音指令处理后 2 秒冷却，避免 TTS 输出的声音被麦克风捡回

---

## 7. 信任与心理模型

传统告警式 DMS 的问题在于：用户感到的是**监视 + 强制**。系统响警报 → 用户感到被评判 → 信任下降。

本系统的设计试图构建另一种心理模型：
- 系统在**观察**但不评判
- 系统在**建议**但不强制
- 用户感到的是**被关注 + 有选择**

用户确认机制不仅是功能需要，更是信任构建机制。每一次"询问 → 确认 → 执行"的完整循环都是一次 micro-trust event：系统证明自己是可预测的、可被取消的、不越界的。

---

## 8. 当前设计的不足

- 哈欠检测延迟：累积 2 次哈欠可能耗时 30 秒至数分钟，用户可能已在等待
- 询问语单一：每次都是相同句式，缺少上下文变化
- 无 neglection 机制：如果用户多次拒绝 Dynamic 建议，系统不应反复询问，但当前未实现此逻辑
- 无法识别用户主动哈欠 vs 自发哈欠：所有哈欠都被视为潜在 cue
