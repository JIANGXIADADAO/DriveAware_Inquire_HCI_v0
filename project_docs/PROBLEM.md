# PROBLEM.md — 项目现存问题总览

> 汇总自：论文评审意见、系统架构文档、交互设计文档、README 中的已知限制。
> 按严重程度排序：**P0 = 阻塞学术发表，P1 = 阻碍工程可用性，P2 = 设计层面待讨论。**

---

## P0 — 学术论证缺陷（无用户数据支撑核心主张）

### 1. 缺少用户实验
论文声称系统"保护 User Agency"、"构建信任"、"降低打扰"，但**没有任何用户研究数据支撑**。
- 无法证明询问式交互确实被用户感知为"有选择"而非"被打扰"
- 无法证明 trust-building 机制（micro-trust event loop）真实发生
- 设计原则停留在纸面论证，属"我觉得这样更好"级别

### 2. 缺少对比基线
没有与以下方案做 A/B 对比：
- 基线 A：纯手动控制（无系统干预）
- 基线 B：自动执行方案（检测到哈欠 → 直接切换模式）
无法得出"询问式优于自动式"的结论，核心贡献无法成立。

### 3. 缺少定量评估指标
以下关键指标全为空白：
- 哈欠检测：precision / recall / F1（无 ground truth 标注）
- 语音识别：WER（Word Error Rate）
- 端到端延迟：从哈欠发生到模式切换完成的总时延
- 意图理解准确率：keyword-only vs. keyword+API 的对比

### 4. 论证逻辑的未检验假设
- **询问本身也是打扰**：语音询问需要驾驶员注意力切换，此成本是否小于自动切换的"剥夺感"？完全未讨论
- **哈欠 = 低唤醒度放松**：文献中哈欠更常关联疲劳/睡眠剥夺，重新定义为"放松线索"需要文献支撑
- **2 次哈欠阈值**：为什么是 2 次？1 次误报太高还是 3 次延迟太大？无数据支撑
- **Rest Mode 默认**：声称"安静优先"，但默认 Rest Mode 本身也是一种预设——系统假定用户不想被打扰，这与"尊重用户意图"的立场存在张力

## P1 — 技术实现问题

### 5. 感知层：参数无调优依据
- MAR 阈值 0.48 从何而来？人群差异、个体差异均未考虑
- Growth rate（~0.06/frame @ 15fps）和 decay factor（0.85）如何确定？
- 衰减模型参考了 AttentionScorer，但未说明与原始模型的差异及原因

### 6. 感知层：单模态脆弱性
- 仅依赖 MAR 单一信号。低光照下 MediaPipe landmark 不稳定（README 已承认）
- 头部姿态变化（侧头、低头）导致 landmark 遮挡时 MAR 失效
- 未融合眼动（PERCLOS）、头部姿态、身体姿态等多模态信号

### 7. 语音层：Whisper 幻读
- Whisper `small` 对短单词误识别严重（"Yes"→"Yinz", "Rest"→"Res"）
- 当前方案是关键词词典扩展（硬编码常见误识别词），属打补丁
- 未考虑 fine-tune、使用 `medium`/`large` 模型、或改用流式 STT 方案
- 幻读导致的关键词命中可能产生误判（如环境噪声被识别为 yes-like 词）

### 8. 语音层：VAD 噪声鲁棒性
- RMS-based VAD 在车内噪声（发动机、风噪、路噪）下可靠性存疑
- 自适应阈值（noise_floor × 3.0）在非稳态噪声（颠簸、鸣笛）下可能失效
- 无噪声环境下的 VAD 性能基准数据

### 9. 架构层：线程通信无消息总线
- 线程间依赖 SharedState + polling（FSM 每帧读取标志位）
- 缺乏事件驱动机制，增加线程/事件类型时复杂度非线性增长
- TTS 完成检测依赖 `tts_done` 标志位，pyttsx3 无精确回调，存在竞态窗口

### 10. 架构层：Whisper 模型加载阻塞主线程
- `stt.load()` 在启动阶段阻塞 5-15 秒（取决于硬件）
- 期间 UI 完全不可交互，无加载进度提示

### 11. 架构层：配置硬编码
- 全部参数在 `config.py` 中硬编码，运行时不可调
- 调试需改代码 + 重启，无法快速迭代参数

## P2 — 设计层面问题

### 12. 询问语的单调性
- 每次询问使用相同的 TTS 句式（"You seem relaxed. Would you like to change to Dynamic Mode?"）
- 缺少上下文变化（如"这是你第三次看起来比较放松了"）
- 多次拒绝后无自适应机制——系统继续以相同方式询问已被重复拒绝的建议

### 13. 缺少 Neglection 机制
- 如果用户多次（如 3 次以上）在相同条件下拒绝 Dynamic 建议，系统应学习并停止询问
- 当前设计：每次 2 哈欠 → 必定触发询问，无论历史拒绝次数
- 这与"非侵入式设计"原则矛盾——反复询问已明确表达偏好的用户是打扰

### 14. 无法区分哈欠类型
- 所有哈欠（疲劳型、放松型、生理型、社交型）被同等对待
- 没有结合上下文（时段、驾驶时长、方向盘行为）区分哈欠含义
- 驾驶员刚上车时的哈欠 vs. 驾驶 3 小时后的哈欠，含义截然不同

### 15. 双路径交互的边界情况
- Path B（持续语音指令）执行时取消 Path A（哈欠触发流程），但取消后的状态恢复逻辑不完整
- 持续语音指令在 `MONITORING` 状态下才接受——如果用户在系统播报 TTS 期间尝试语音指令会被忽略
- 2 秒 cooldown 是固定值，未根据实际 TTS 长度动态调整

## P3 — 实验环境问题

### 16. 桌面原型与真实座舱的差距
- Pygame 桌面模拟：无方向盘、无驾驶任务、无真实噪声
- 驾驶是视觉-手动-认知三重任务，当前环境完全剥离了驾驶负荷
- 摄像头 15fps 固定角度，真实驾驶中头部运动、光照剧烈变化

### 17. 多任务干扰未评估
- 语音交互虽占用资源较少，但在驾驶负荷下并非零成本
- 系统询问 → 驾驶员理解 → 决策 → 语音回应的总认知负荷未被测量
- 驾驶任务与语音交互的互相干扰程度未知

```
  P0-1,2,3  用户实验（≥20 人）+ 对比基线 + 定量指标
  P0-4      论证逻辑补全（trade-off 讨论 + 文献支撑）

  P3-16,17  驾驶模拟器验证

长期（工程落地）：
  P1-9,10,11  架构重构（消息总线 + 异步加载 + 热配置）
  P2-14        哈欠多模态上下文感知
  P3-16        实车环境部署
```
