# 主动式座舱交互 — 人机交互研究原型

> **如果座舱不是自动执行，而是先问一声呢？**
>
> 本原型探索智能座舱 HCI 中一个简单但未被充分审视的想法：系统感知驾驶员的生理线索，以语音发起询问，由驾驶员做出决定。不自动接管，不报警，只是一句询问。

---

## 研究问题

当前驾驶员监测系统（DMS）面临一个结构性问题：**生理信号本身具有模糊性，但系统必须对之做出反应。** 一次哈欠可能意味着疲劳、舒适、饭后困倦，也可能只是伸展身体。当系统基于此类模糊信号自动执行——切换驾驶模式、触发警报、调节座舱——它既可能误读驾驶员状态，更关键的是，**剥夺了驾驶员的决策权。**

这引出了一个核心 HCI 问题：

> *能否将座舱交互从"指令式自动化"转向"询问式协作"——系统承担感知与建议，决策权始终在人？*

---

## 交互哲学

### 询问而非自动执行

系统不自动执行操作。当检测到有意义的线索（如累积哈欠暗示低唤醒度放松状态），它主动询问：

> *"您看起来比较放松。需要切换到 Dynamic Mode 吗？"*

驾驶员说"要"或"不要"。至此完成。系统承担**感知 + 建议**的职责；人类保留**决策 + 确认**的权力。这是 **Shared Control（共享控制）** 从方向盘延伸到整个座舱环境的具体实践。

### 哈欠是线索，不是警报

传统 DMS 将驾驶员状态线索——尤其是哈欠——视为需要立即干预的确切危险信号。我们认为这是一个范畴错误。哈欠是**情境依赖的、多因的生理事件**，而非疲劳警报。将其视为线索而非触发条件，意味着系统*询问*而非*擅自行动*，通过对话而非猜测来消解信号模糊性。

### 保护人类决策权

每个状态转换都由明确的用户行为门控：
- 系统在**观察**，但不评判
- 系统在**建议**，但不强制
- 用户感到的是**被关注**而非**被监视**，是**有选择**而非**被覆盖**

每次完整的交互循环——观察 → 询问 → 确认 → 执行——都是一次 **micro-trust event（微型信任事件）**：系统证明自己是可预测的、可被取消的、不越界的。

### 默认安静

Rest Mode 是系统默认状态。除非积累充分证据（Rest Mode 下 ≥2 次哈欠），系统保持沉默。这体现了一个设计立场：**安静优先于打扰，维持现状优先于强制变更。**

## 工作原理（概念层面）

```
 驾驶员状态线索           系统询问                  驾驶员回应              结果
 (检测到哈欠)     →    "需要切换模式吗？"     →    "要" / "不要"     →    切换 / 保持
```

### 两条交互路径

| 路径 | 触发 | 方向 | 示例 |
|---|---|---|---|
| **Path A** | 累积哈欠线索（Rest Mode 下 ≥2 次） | 系统发起 | 系统察觉放松 → 询问切换 |
| **Path B** | 持续语音指令 | 驾驶员发起 | 驾驶员随时说 *"切换到 dynamic mode"* |

Path B 始终优先于 Path A。驾驶员的明确指令覆盖系统发起的任何询问流程。Path A 在两次意图识别失败后静默退回监测——避免常见助手的重复追问故障模式。

### 共享控制循环

系统运行 8 状态非阻塞循环：Monitoring → Yawn Detected → Inquiring → Listening → Transcribing → Parsing → Confirming → Switching → 回到 Monitoring。每个状态转换仅由用户相关事件门控（哈欠检测、语音结束、意图解析），而非系统内部计时器或阈值。

### 模式概念

| | Rest Mode *(默认)* | Dynamic Mode |
|---|---|---|
| 环境光 | 冷蓝 | 暖橙 |
| 音效 | 舒缓 | 高节奏 |
| 虚拟空调 | 26°C / 弱风 | 18°C / 强风 |
| 含义 | 驾驶员偏好当前舒适 | 驾驶员需要更警醒的环境 |

两种模式不仅是"运动 vs 舒适"的二选一。它们代表了两种**用户意图路径**：接受系统的主动建议（Dynamic），或拒绝并维持现状（Rest）。

## 技术实现（概要）

**感知**：摄像头 → MediaPipe 面部 478 特征点 → 嘴部纵横比（MAR）→ 指数衰减评分模型（非简单连续帧计数；短暂特征点抖动不重置累积证据）。

**语音管道**：pyttsx3 TTS 播报询问 → 5 秒麦克风录音 → Whisper `small` STT 转写 → 两层 NLP 意图解析（关键词匹配 0ms 延迟；DeepSeek API 作为 fallback）。

**状态管理**：线程安全的共享状态 + 8 状态状态机，由 Pygame 主循环以 60fps 轮询。独立线程处理摄像头采集、TTS 播报、持续语音监听。

**跨平台说明**：当前仅在 Windows 测试（pyttsx3 使用 SAPI5）。macOS（NSSpeechSynthesizer）和 Linux（espeak）后端可用但未经测试。

## 快速开始

```bash
pip install -r requirements.txt
# 将 MediaPipe face_landmarker.task 放入 assets/
# （可选）在 .env 中设置 DEEPSEEK_API_KEY
python main.py
```

键盘快捷键：`1` Dynamic, `2` Rest, `Y`/`Z` 模拟哈欠, `Esc` 退出.

## 当前局限（研究范围声明）

这是一个**桌面研究原型**，非产品级系统。已知不足：

- **尚未进行用户实验** — 关于信任、主体性、非侵入性的设计主张尚未经实证验证
- **单模态感知** — 仅用 MAR；眼动、头部姿态、心率等尚未融合
- **固定阈值** — 未做个体差异标定
- **桌面模拟** — 无真实驾驶任务、无方向盘、无车内噪声环境
- **Whisper `small`** 对短单词偶有误识别；通过关键词词典扩展缓解

完整改进路线图见 [PROBLEM.md](project_docs/PROBLEM.md)。
